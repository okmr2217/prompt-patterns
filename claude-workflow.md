# Claude Code ワークフローガイド

AI（Claude Code）と協業する開発プロジェクトで使用する、セッション管理・バージョニング・Git ワークフローの設計まとめ。
他プロジェクトへの適用・テンプレートとして使用することを想定している。

---

## 目的

- AI セッションをまたいで文脈を維持する
- 作業の再現性・引き継ぎ可能性を高める
- バージョン管理と変更履歴を一貫して運用する
- どのプロジェクトでも同じパターンで適用できる

---

## ファイル構成

```
project-root/
├── CLAUDE.md              # Claude への指示書（ルール＋プロジェクト固有情報）
├── CHANGELOG.md           # ユーザー向け変更履歴
└── docs/
    ├── project.md         # プロジェクト安定情報（コンセプト・技術設計・アーキテクチャ）
    ├── handoff.md         # セッション間の引き継ぎ（現在地・積み残し・次にやること）
    ├── session-log.md     # セッション作業記録（時系列降順）
    └── versioning.md      # バージョン更新プロセスガイド（Claude 参照用）
```

### 最低限のセット（小規模プロジェクト）

```
project-root/
├── CLAUDE.md       # 必須
├── CHANGELOG.md    # バージョン管理するなら必須
└── docs/
    └── handoff.md  # セッションをまたぐなら必須
```

`project.md`・`session-log.md`・`versioning.md` は規模・運用に応じて追加する。

---

## 各ファイルの役割

### `CLAUDE.md` — Claude への指示書

Claude が毎回参照するプロジェクト憲法。**2ゾーン構成**にすることで再利用性を高める。

```
# [プロジェクト名]

---

<!-- ▼ プロジェクト固有（このプロジェクト専用の設定） ▼ -->

## Tech Stack
## コマンド
## コーディングルール
## プロダクト前提
## デザインシステム
## やらないこと       ← 禁止事項を明示する

---

<!-- ▼ 汎用ルール（他プロジェクトでも同じ） ▼ -->

## Git ワークフロー
## セッション管理

---

## 参照ドキュメント
```

**ポイント：「やらないこと」を明示する**

Claude はデフォルトで「良かれと思って」リファクタリング・コメント追加・抽象化を行う。
禁止事項を `CLAUDE.md` に明記することで、意図しない変更を防ぐ。

```md
## やらないこと

- 不要な抽象化・ライブラリ追加
- コードコメント・docstring の追加（変更していないコードへ）
- エラーハンドリングの過剰追加（起こりえないケースへの対処）
- リファクタリング・整理（明示的に依頼されていない場合）
```

---

### `docs/project.md` — プロジェクト安定情報

変化が少ない情報を集約する。`handoff.md` の肥大化を防ぐ目的。

**含める情報:**
- プロダクトコンセプト・思想
- コア概念の定義
- 技術スタック
- アーキテクチャ・鉄則
- フォルダ構成
- DB スキーマと設計判断
- デザインシステム

**更新頻度:** 低い（技術スタック変更・設計変更時のみ）

---

### `docs/handoff.md` — セッション引き継ぎ（現在地）

次の AI セッションが開始時に読む「今どこにいるか」ドキュメント。

**含める情報:**
- 現在のバージョン・最終更新日（ヘッダーに明記）
- 現在の実装状態（画面・機能一覧）
- 積み残し・注意点（前セッションで発生した課題）
- 今後の候補（短期・中期・長期）
- 次のセッションで相談したいこと

**運用:**
- セッション開始時に読む
- セッション終了時に必ず更新する
- `project.md` と重複する情報は書かない

**`project.md` との役割分担:**

| | project.md | handoff.md |
|---|---|---|
| 内容 | 「何を作っているか」 | 「今どこにいるか」 |
| 更新頻度 | 低い | 毎セッション |
| 例 | コンセプト・スタック | 実装状態・積み残し |

---

### `docs/session-log.md` — セッション作業記録

セッション終了時に追記する作業ログ。新しい記録を**先頭に追記（降順）**。

**1セッションの記録フォーマット:**

```md
---

## YYYY-MM-DD

### やったこと
- 実装・修正の概要

### 改善案（未対応）
- 気づいたが今回は対応しなかったもの

### 失敗したアプローチ
- 次に同じ轍を踏まないための記録

### 技術メモ
- ハマったこと・解決策

### 次にやりたいこと
- 優先度付きで
```

**ポイント:** `handoff.md` は「今の状態」、`session-log.md` は「どう来たか」。役割を分離することで `handoff.md` を常に最新・簡潔に保てる。

---

### `docs/versioning.md` — バージョン更新ガイド（Claude 参照用）

新バージョンを追加するときに Claude が参照するプロセスガイド。詳細は `docs/versioning.md` を参照。

**基本方針:**
- バージョン番号の正本は `package.json` の `version`（他ファイルで重複管理しない）
- 変更履歴は `CHANGELOG.md` に記録（ユーザー視点で書く）
- 新バージョン追加時は必ず Git 履歴・差分を確認してから書く（推測禁止）

---

### `CHANGELOG.md` — 変更履歴

ユーザー向け変更履歴。技術詳細ではなくプロダクト上の変更を書く。

```md
# Changelog

## x.y.z - YYYY-MM-DD

- 変更内容1（ユーザー視点）
- 変更内容2
```

```
  └── コンテキスト 60% 到達時：session-log.md + handoff.md を更新して /compact

セッション終了
  ├── 1. typecheck && lint を実行
  ├── 2. 問題なければコミット
  ├── 3. session-log.md の先頭にセッション記録を追記
  └── 4. handoff.md を更新（実装状態・積み残し・次回相談事項）
```

## Git ワークフロー

### コミットメッセージ形式

```
<prefix>: <変更内容を日本語で>
```

### プレフィックス一覧

| プレフィックス | 用途 |
|---|---|
| `feat:` | 新機能追加 |
| `fix:` | バグ修正 |
| `refactor:` | リファクタリング（機能変更なし） |
| `chore:` | ビルド・設定・依存関係の変更 |
| `docs:` | ドキュメント変更 |
| `test:` | テスト追加・修正 |
| `style:` | フォーマット・スタイルのみの変更 |

### ルール

- コミット前に typecheck && lint を実行
- 1つの論理的変更 = 1コミット
- コミットメッセージは日本語（プレフィックスは英語）

---

## 他プロジェクトへの適用手順

1. **`CLAUDE.md` を作成** — 汎用ルール部分（Git ワークフロー・セッション管理）はそのまま、プロジェクト固有部分を書き換える
2. **`docs/project.md` を作成** — プロジェクトのコンセプト・技術設計を記載
3. **`docs/handoff.md` を作成** — 初期状態（実装済み機能・積み残し）を記載
4. **`docs/session-log.md` を作成** — `# Session Log` のヘッダーのみで始める
5. **`docs/versioning.md` をコピー** — プロジェクト固有の記述があれば更新
6. **`CHANGELOG.md` を作成** — `# Changelog` のみで始める
7. **`CLAUDE.md` の参照ドキュメントセクション** を確認してパスが正しいか確認する

## 設計の背景・思想

### なぜ `handoff.md` が必要か

Claude のセッションはコンテキストウィンドウ内でしか記憶を持たない。セッションをまたいだ継続的な開発では、「今どこにいるか」を毎回伝える必要がある。`handoff.md` はその起点となる文書。

### なぜ `project.md` を分けるか

`handoff.md` が肥大化すると、毎回更新が必要な「現在地」と、ほぼ変わらない「プロジェクト設計」が混在してしまう。分離することで `handoff.md` を常に最新・簡潔に保てる。

### なぜ `session-log.md` を分けるか

`handoff.md` は「今の状態」、`session-log.md` は「どう来たか」。失敗したアプローチや技術メモは `session-log.md` に残し、`handoff.md` を肥大化させない。

### なぜバージョン番号を `package.json` に正本を置くか

複数ファイルに分散させると同期ズレが起きる。表示箇所（UI）は `package.json` を参照するだけにすることで、更新箇所を1つに絞る。

### なぜ Claude に禁止事項を明示するか

Claude はデフォルトで「良かれと思って」リファクタリングしたり、不要な抽象化を追加したりする。`CLAUDE.md` の「やらないこと」セクションに明記することで、意図しない変更を防ぐ。
