# パリッと開発日記 ブログ記事ライティングガイド

このドキュメントは、Claude Code で「パリッと開発日記」のブログ記事を執筆するときのガイドです。
記事の種類を問わず、このブログにふさわしい記事を書くための基準とルールをまとめています。

---

## このガイドの使い方

1. 記事を書く前に、このガイド全体を読む
2. 「記事タイプ別の構成」から該当するタイプを選ぶ
3. 「共通ルール」に従って書く
4. 書き終わったら「チェックリスト」で確認する

---

## ブログの基本情報

- **ブログ名**: パリッと開発日記
- **著者**: okmr（個人開発者）
- **ブログリポジトリ**: okmr2217/paritto-dev-diary
- **記事の配置先**: `content/posts/[slug].md`
- **画像の配置先**: `public/posts/[filename].png`
- **想定読者**: 技術ブログの読者、個人開発に興味がある人、一般ユーザー

## 共通ルール

### トーンとスタイル

- 自然な日本語で書く
- 少しカジュアルだけど、砕けすぎない
- 開発者ブログらしい、等身大のトーン
- 誇張しない。「革命的」「最強」みたいな大げさな表現はNG
- 専門用語は最小限。使う場合は簡単に補足する
- 絵文字は見出しの装飾（📅、🔍 など）に限定的に使う程度。本文中には使わない

### 文章の書き方

- 一文は短めに。体言止めも適度に使う
- 「〜〜ということです。」のような冗長な締め方は避ける
- 読者に語りかけるような導入を意識する（「〜〜ありませんか？」「〜〜ですよね。」）
- 段落の切り替えで余白を作る。詰め込みすぎない
- 箇条書きと文章を適切に使い分ける。箇条書きだけで記事を構成しない

### 構造

- frontmatter は必須（title, date, description, tags）
- description はコンセプトや一番伝えたいことを1〜2文で書く
- 見出しは `##` から使う（`#` は使わない）
- 機能紹介やステップ紹介は `###` で区切る
- 導入 → 本題 → まとめ の流れを基本にする

### 画像

- 記事内で画像を使う場合は `![alt text](/posts/filename.png)` 形式
- alt text は画面の内容を簡潔に書く
- 画像は読者の理解を助ける目的で入れる。装飾目的では入れない

### 文字数

- 目安は 2000〜3000 文字程度
- 短くてもよいが、薄い内容を水増ししない
- 長くなる場合は「目次（## 目次）」を入れる

### 技術的な内容について

- 技術スタック、設計上のこだわり、コードの詳細は**書かない**のが基本
- ただし記事のテーマ自体が技術的な内容（MCPサーバーの紹介など）の場合は、読者に伝わるレベルで最小限に書く
- 「何の技術を使ったか」ではなく「それによってどんな体験が生まれたか」を書く

## 記事タイプ別の構成

### タイプA: アプリ紹介

自作アプリの公開を報告し、どんなアプリかを紹介する記事。

**構成**:
1. **はじめに** — 公開の報告 + アプリの一言説明 + 作った背景や動機
2. **コンセプト / 特徴** — どんな思想で作ったか、他との違い、提供する価値
3. **主な機能**（### で各機能を紹介）— スクリーンショットつきで、体験ベースで説明
4. **どんな人におすすめか** — 想定ユーザーを具体的に
5. **今後の予定** — 未実装の機能のみ。実装済みと混同しない
6. **おわりに** — アプリの価値を一言で再提示して締める

**参考記事**: Yarukoto（導入〜概要の流れ）、Peak Log（機能紹介セクション）

**注意事項**:
- 実装と仕様に基づいた内容にすること。書く前に spec.md、claude.md、実装コード、UI文言を確認する
- 存在しない機能は書かない
- 未実装機能は「今後の予定」セクションに明確に分ける
- UI上の用語（ボタン名、画面名、概念名）は実装に合わせる

### タイプB: ツール / サービス紹介

MCP サーバーなど、アプリ以外の自作ツールを紹介する記事。

**構成**:
1. **はじめに** — ツールの概要 + 何ができるかを端的に
2. **コンセプト** — なぜこのアプローチを取ったか
3. **なぜ作ったのか** — 課題や動機
4. **主な機能 / ツール**（### で各機能を紹介）— 「使い方のイメージ」をコードブロックなどで具体的に見せる
5. **どんな人におすすめか**
6. **これからの予定**
7. **まとめ**

**参考記事**: furikaeri-mcp

### タイプC: ハウツー / 体験記

やってみた系、試してみた系の記事。

**構成**:
1. **導入** — 読者の共感を引く問いかけや状況説明 + 結論の先出し
2. **ステップ1〜N**（### でステップを区切る）— 実際にやったことを時系列で
3. **結果 / 成果** — 具体的に何ができたか
4. **良かった点・いまいちだった点** — 正直に書く
5. **おわりに** — まとめと読者へのおすすめ

**参考記事**: YouTube Music DJ

### タイプD: エッセイ / 個人的な記録

開発とは直接関係のない、個人的な体験や考えを書く記事。

**構成**: 自由。ただし時系列で書くことが多い。

**注意事項**:
- このタイプでは構成の型にはめなくていい
- 自分の言葉で、正直に書く
- 読者に何かを教える必要はない

**参考記事**: hospitalization

## frontmatter テンプレート

```yaml
---
title: "記事タイトル"
date: "YYYY年M月D日"
description: "記事の概要を1〜2文で"
tags: ["タグ1", "タグ2"]
---
```

### title のルール

- タイプA: 「[アプリ名]を公開しました」「[目的]のために[アプリ名]を作りました」
- タイプB: 「[できること]を[方法]で実現する[ツール名]を自作した」
- タイプC: 「[やったこと]をしたら、[結果]になった」
- 共通: 読んで内容がわかるタイトルにする。煽りすぎない

## 記事を書く前の準備

記事を書き始める前に、以下を確認・整理してください。

### アプリ / ツール紹介の場合

1. **リポジトリを確認する**
   - spec.md / claude.md / README.md を読む
   - 実装コードを確認する（画面構成、UIの文言、実装済み機能）
   - 必要に応じてコミット履歴を確認する

2. **整理する**
   - このアプリ / ツールの価値は何か？
   - 読者にどの体験を伝えるべきか？
   - 実装済みの機能は何か？未実装は何か？

### ハウツー / 体験記の場合

1. **事実を整理する**
   - 何をやったのか（手順）
   - どんな結果になったのか
   - 良かったこと、いまいちだったこと

2. **伝えたいことを決める**
   - この記事を読んだ人に何を持ち帰ってほしいか？

## チェックリスト

記事を書き終えたら、以下を確認してください。

### 全記事共通

- [ ] frontmatter（title, date, description, tags）が埋まっている
- [ ] description がコンセプトや要点を端的に伝えている
- [ ] 導入で「この記事は何の話か」がすぐわかる
- [ ] 誇張表現がない
- [ ] 専門用語に補足がある（またはそもそも使っていない）
- [ ] 箇条書きだけの段落がない（文章と混ぜる）
- [ ] 画像のパスが正しい形式になっている
- [ ] 読み返して冗長な部分を削った

### アプリ / ツール紹介

- [ ] 実装と仕様に基づいている（想像で書いていない）
- [ ] 存在しない機能を書いていない
- [ ] 未実装機能が「今後の予定」に分かれている
- [ ] UI文言・概念名が実装と一致している
- [ ] 機能の説明が「何ができるか」ではなく「どんな体験か」になっている
- [ ] 技術スタックや設計の詳細を書いていない（テーマ上必要な場合を除く）

### ハウツー / 体験記

- [ ] 手順が具体的で再現可能
- [ ] 良い点だけでなく、いまいちだった点も正直に書いている
- [ ] 結論が先に出ている（最後まで読まないとわからない構成になっていない）
